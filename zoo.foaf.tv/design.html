Design notes

Status: most of this document should be semi-automatically generated. For now focus is on proving the infrastructure (testing RDFa parser support), cleaning and studying the vocabulary contents. A few terms are documented here manually, to make sure we have things working right. This list is a rough todo list. Note that the initial integration is purely regarding name clashes; there are of course opportunities for closer collaboration amongst these vocabularies, eg. DC's notion Location with geonames, geo or foaf:based_near. That would be a much larger project.

RDFa 1.1 parsers that can handle this include the Ruby gem rdf-rdfa and pyRDFa.

  1. We much prefer properties to begin lowercase, classes uppercase. One less thing for users to remember.
  2. We begin with Dublin Core (dcterms:) and all the non-archaic FOAF terms, to describe people, groups and organizations and the things they make and do.
  3. We add the Organization vocabulary, which expands on FOAF to model organizational structures more explicitly.
  4. We add the SIOC vocabulary, providing a model for various online discussion systems.
  5. We add Creative Commons vocabulary for talking about content, copyright and licensing.
  6. We add all W3C's main technical 'built-ins': RDF, RDFS and OWL terminology.

Name conflicts and other decisions (selected highlights...)

  • bibo:Collection and skos:Collection; we default to skos:Collection (consulted Bruce Darcus re bibo: on this). Also swanco:Collection.
  • foaf:member, mo:member and org:member; since mo:member specializes foaf:member, we default to foaf:member (Dave Reynolds confirmed this makes sense re org:).
  • dcterms:title and foaf:title; we default to Dublin Core's dcterms:title. Considering aliasing foaf:title as 'honorificPrefix' to match Portable Contacts.
  • dcterms:Agent and foaf:Agent; we default to FOAF Agent, noting that the vocabularies declare reciprocal mappings already.
  • foaf:maker is equivalent to dc:creator; we defer to the Dublin Core version.
  • rdfs:Class and owl:Class and geonames:Class; we defer for now to rdfs:Class, but are open to suggestions re OWL and RDFS.
  • gr:Brand and po:Brand - investigating, seems gr:Brand is a superclass; and sensible default?
  • swanci:Article and bibo:Article; ditto 'Book'; default to Bibo as more general purpose? Not clear we can fit all of SWAN vocabs in here anyway.
  • bibo:Code, geonames:Code - todo: investigate relative usage levels and importance within the vocabulary. Do meanings overlap?
  • Conclusion re SWAN - these vocabs bring too many name clashes for now, so we won't reflect terms into defaults without more careful analysis.
  • sioc:Container, rdfs:Container. Default to SIOC since RDFS's is just superclass of Bag, Seq, Alt and rarely used in normal instance data.
  • For more raw details, see manifest
  • foaf:Document and bibo:Document; assume they're the same. Defaulting to foaf:Document for now. See previous discussions. Ditto :Image class.
  • bibo:Email and vcard:Email - todo: investigate
  • bio:Event and ev:Event - same notion of events? See also other event ontologies. Unclear what to do here.
  • mo:Genre " An expressive style of music." and po:Genre "Genre - An anchor point for a programmes' genre taxonomy, e.g. 'Drama'/'Biographical'.". Awkward, both are very specific. Todo: investigate with YvesR.
  • bio:Inverval and tl:Interval - todo: compare semantics and data structure.
  • mo:Label and vcard:Label; default to Music Ontology.
